# 体系的デバッグ

**鉄則: 根本原因の調査なしに修正を行ってはならない。**

## フェーズ1: 根本原因の調査
1. エラーメッセージを注意深く読む（スタックトレース、行番号、エラーコード）
2. 一貫した再現手順を確立
3. `git diff`・最近のコミット・環境差異を確認
4. マルチコンポーネントの場合: 各境界でログ出力→証拠収集→故障箇所特定
5. データフロー追跡: 不正値の発生源まで遡る

## フェーズ2: パターン分析
- 同コードベース内の動作する実例を探す
- リファレンス実装と比較（全行読む）
- 動作するものと壊れたものの差異を全てリストアップ

## フェーズ3: 仮説と検証
- 単一の仮説を明確に述べる
- 最小限の変更で検証（一度に一つの変数だけ）
- 失敗→新仮説。修正を重ねない

## フェーズ4: 実装
1. 失敗するテストケースを作成
2. 根本原因に対する単一の修正
3. テスト通過を確認
4. **3回以上修正失敗 → 停止してアーキテクチャを疑う → ユーザーと議論**

## 危険信号（停止してフェーズ1に戻れ）
「とりあえず応急処置」「多分Xだろう」「完全には理解してないが」「もう一回試そう」
