# リサーチ付きコンテンツ執筆支援

リサーチ・アウトライン・下書き・推敲を支援するライティングパートナー。ユーザー固有の文体を維持しながら協働する。

> 元リポジトリ: https://github.com/composio-skills/content-research-writer

## 発動条件

- ブログ記事・記事・ニュースレターの執筆
- 教育コンテンツやチュートリアルの作成
- ソートリーダーシップ記事の起草
- 引用・参考文献付きの執筆
- フック（導入部）の改善依頼
- セクション単位のフィードバック依頼

## 機能一覧

1. **協働アウトライニング**: アイデアを一貫した構成に整理
2. **リサーチ支援**: 関連情報の調査と引用追加
3. **フック改善**: 冒頭部を注意を引く内容に強化
4. **セクション別フィードバック**: 執筆中にセクション毎のレビュー
5. **文体保持**: ユーザー固有のトーンとスタイルを維持
6. **引用管理**: 参考文献の追加とフォーマット
7. **反復的推敲**: 複数ドラフトを通じた改善

## 基本ワークフロー

1. **アウトラインから開始**: 「[トピック]の記事のアウトラインを作って」
2. **リサーチと引用追加**: 「[テーマ]をリサーチして引用を追加して」
3. **フック改善**: 「導入部をもっと魅力的にして」
4. **セクション別フィードバック**: 「このセクションをレビューして」
5. **推敲と仕上げ**: 「全体の流れ・明瞭さ・一貫性をレビューして」

## 手順詳細

### 1. 執筆プロジェクトの理解

以下を確認する:
- トピックと主張は何か？
- ターゲット読者は誰か？
- 希望する長さ/形式は？
- 目的は？（教育、説得、解説、エンタメ）
- 既存のリサーチやソースはあるか？
- 文体は？（フォーマル、会話的、技術的）

### 2. 協働アウトライニング

```markdown
# 記事アウトライン: [タイトル]

## フック
- [冒頭の一文/ストーリー/統計]
- [読者が読むべき理由]

## 導入
- 背景と文脈
- 問題提起
- 記事の概要

## 本文セクション

### セクション1: [タイトル]
- 要点A、要点B
- 例/根拠
- [リサーチ必要: 具体的トピック]

### セクション2: [タイトル]
- 要点C、要点D
- データ/引用が必要

## 結論
- 要点まとめ
- 行動喚起
- 締めの一言

## リサーチToDo
- [ ] [トピック]のデータを探す
- [ ] [概念]の事例を取得
- [ ] [主張]の出典を確認
```

### 3. リサーチ実施

出力例:
```markdown
## リサーチ: AIの生産性への影響

主な知見:
1. **生産性向上**: コンテンツ作成タスクで40%の時間短縮 [1]
2. **導入率**: ナレッジワーカーの67%が毎週AIツールを使用 [2]

引用:
[1] McKinsey Global Institute. (2024). "The Economic Potential of Generative AI"
[2] Stack Overflow Developer Survey (2024)
```

### 4. フック改善

ユーザーの導入部を分析し強化する。

**分析項目**: 良い点、改善余地、感情的インパクト

**改善案を複数提示**:

案1（データ駆動型）:
> 「先月、AIに500件の顧客インタビューを分析させた。3週間かかる作業が30分で終わった。」

案2（質問型）:
> 「全顧客と話し、全レビューを読み、全サポートチケットを分析できたら？」

案3（ストーリー型）:
> 「佐藤さんは2週間かけて間違った機能を作った。データ処理が追いつかなかったからだ。」

**チェック項目**: 好奇心を刺激するか？ 価値を約束するか？ 具体的か？ 読者に合っているか？

### 5. セクション別フィードバック

```markdown
# フィードバック: [セクション名]

## 良い点
- [強み1]、[強み2]

## 改善提案

### 明瞭性
- [具体的な問題] → [修正案]

### 流れ
- [遷移の問題] → [改善案]

### 根拠
- [裏付け必要な主張] → [引用や事例の追加]

### スタイル
- [トーンの不一致] → [文体に合わせた修正]

## 具体的な行レベル編集

原文:
> [ドラフトからの引用]

修正案:
> [改善版]

理由: [説明]
```

### 6. 文体保持の原則

- **スタイルを学ぶ**: 既存の文章サンプルを読む
- **提案する、置き換えない**: 選択肢を提示、指示はしない
- **トーンを合わせる**: フォーマル/カジュアル/技術的/フレンドリー
- **選択を尊重する**: ユーザーが自分の版を好めば支持する
- **強化する、上書きしない**: より良くする、変えるのではない

定期的に確認: 「これはあなたらしい表現ですか？」「トーンは合っていますか？」

### 7. 引用管理

**インライン引用**:
```markdown
研究では40%の生産性向上が示された（McKinsey, 2024）。
```

**番号付き参照**:
```markdown
研究では40%の生産性向上が示された [1]。

[1] McKinsey Global Institute. (2024)...
```

### 8. 最終レビュー

```markdown
# 全体レビュー

## 総合評価
- 強み、インパクト評価

## 構造と流れ
- 構成、遷移の質、ペーシング

## コンテンツ品質
- 論拠の強さ、根拠の十分さ、事例の効果

## 技術品質
- 文法、一貫性、引用の完全性

## 公開前チェックリスト
- [ ] 全主張に出典あり
- [ ] 引用フォーマット済み
- [ ] 事例が明確
- [ ] 遷移がスムーズ
- [ ] 行動喚起あり
- [ ] タイポチェック済み
```

## 執筆ワークフロー別ガイド

### ブログ記事
アウトライン → リサーチ → 導入執筆+FB → 本文セクション+FB → 結論 → 推敲

### ニュースレター
フックのアイデア → 簡易アウトライン → 一気に下書き → 明瞭性とリンク確認 → 仕上げ

### 技術チュートリアル
手順アウトライン → コード例 → 説明追加 → 手順テスト → トラブルシューティング → 精度確認

### ソートリーダーシップ
独自の切り口 → 既存視点リサーチ → テーゼ構築 → 強いPOVで執筆 → 根拠追加 → 結論

## ベストプラクティス

### リサーチ
- 引用前にソースを検証する
- 可能な限り最新データを使用する
- 異なる視点をバランスよく取り入れる

### フィードバック
- 具体的に求める:「技術的すぎる？」
- 懸念を共有:「このセクションは冗長かも」
- 代替案を依頼:「別の説明方法は？」

### 文体
- 自分の文章サンプルを共有する
- トーンの好みを指定する
- 「それは自分らしい！」と良い一致を指摘する
