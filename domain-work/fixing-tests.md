# テスト修正の体系的アプローチ

## フロー
1. **全テスト実行** → 失敗を特定
2. **グルーピング**: エラー種別/モジュール/根本原因で分類
3. **優先順位**: 影響テスト数が多いもの＆インフラ系を優先
4. **修正→検証→次グループ** を繰り返す
5. **最終検証**: 全テストスイート実行

## 修正順序
| 優先度 | カテゴリ | 例 |
|--------|---------|-----|
| 1 | インフラ | ImportError、依存欠落、設定 |
| 2 | API変更 | シグネチャ変更、モジュール再編 |
| 3 | ロジック | アサーション失敗、エッジケース |

## ルール
- 1グループずつ修正、通過確認してから次へ
- 変更は最小限
- `git diff`で最近の変更を把握
- 最終的に全テストスイート通過を確認
